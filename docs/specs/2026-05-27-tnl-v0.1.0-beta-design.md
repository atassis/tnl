# tnl v0.1.0-beta — Design

Status: approved by user 2026-05-27, ready for implementation plan.

## Purpose

v0.1.0-alpha shipped to your-gateway and proved the architecture end-to-end (see
`2026-05-26-tnl-design.md` and `2026-05-26-tnl-v0.1.0-alpha.md`). It is usable
but rough: the binary list is two executables, the only way to install is to
build from source, the only way to hand a client a credential is "ssh into the
server, cat the token file". That floor is too high for anyone but us.

The beta widens the audience from "the author" to "any developer with a Linux
server and 10 minutes". Three things change:

1. **First-run UX.** Three installers — `install.sh` (client), `tnld-install.sh`
   (server), and a `tnl init` / `tnld init` interactive wizard — get a fresh
   user from zero to a working tunnel without manual config editing or token
   copy-paste.
2. **Pairing protocol.** A short-lived pairing code replaces the
   read-token-from-file flow, so plaintext bearer tokens never travel through
   the user's shell history, Slack, or clipboard.
3. **Daily-driver feature set.** Inspector, `tnl status`/`stop`, admin token
   CLI, reconnect with reattach window, healthcheck subcommand, random
   subdomain — all features the spec already promised in §4 of the original
   design but were deferred to alpha+.

Plus a full CLI hygiene pass (XDG paths, `--json`, NO_COLOR, completions, man
pages, structured exit codes, `tnl doctor`) and OSS publication infrastructure
(CI, GitHub Releases, ghcr.io image, semver policy, CHANGELOG, contributing
guide). Targeted scope is "Medium" — the IP allow-list, basic auth on
tunnels, rate limiting on the data plane, and metrics endpoint are deferred to
v0.1.0-rc.1.

The audience choice is **public OSS** (not "me + friends"): repo is public,
releases are signed and downloadable, install.sh works on Linux + macOS, the
deploy layer is generic — the operator supplies their own domain and host. This implies
backward-compat care, semver discipline, and CONTRIBUTING.md from day one.

## Out of scope (deferred to v0.1.0-rc.1 or v0.2)

- `--auth USER:PASS` HTTP basic auth on tunnels (rc.1)
- `--allow CIDR` IP allow-list on tunnels (rc.1)
- Per-tunnel rate limiting on data plane (rc.1)
- Prometheus `/metrics` endpoint (rc.1)
- Homebrew formula, apt/rpm packages (rc.1 or 1.0)
- TCP / UDP tunnels (`tnl tcp`) (v0.2)
- Web UI Inspector — terminal-only by deliberate choice
- DNS provider automation (Cloudflare API record creation, etc.) — docs only
- VPS bootstrap (apt install docker, etc.) — assumed prerequisite
- Cosign / sigstore signed releases (1.0)
- Multi-host failover (never; out of scope for this project)

### Backward compatibility

v0.1.0-alpha clients must continue working against v0.1.0-beta daemon for at
least one full beta cycle. The wire protocol is extended via additive
`ControlMsg` variants and new optional fields (serde tagged enums + `Option`).
Existing `CreateTunnelReq { subdomain: String }` becomes
`CreateTunnelReq { subdomain: Option<String> }` — alpha clients that always
sent `subdomain` continue to send `Some(...)`, daemon behaviour is unchanged.

REST endpoints are additive. Existing `/healthz`, `/whoami`, `/control` stay
identical.

Config TOML is additive — new optional keys, no rename of existing ones.

## Components

### Pairing protocol

Replaces the "user copies plaintext token from server file" handshake. The
threat model: the bearer token (long-lived credential) must never travel
through clipboard / Slack / shell history. The pairing code is a short-lived
(default 5 minutes) one-time secret that is safe to transmit verbally or in
chat because it expires fast and is rate-limited at the redeem endpoint.

**Wire types** (in `tnl-protocol::messages`):

```rust
pub struct PairCreateReq {
    pub name: String,            // tokens.toml entry name
    pub expires_in_sec: u32,     // server clamps to [60, 900]
}

pub struct PairCreateResp {
    pub code: String,            // "AB-12-CD"
    pub expires_at: SystemTime,
    pub invite_url: String,      // "https://<endpoint>/invite/AB-12-CD"
}

pub struct PairRedeemReq {
    pub code: String,            // normalized: uppercase, no dashes/spaces
}

pub struct PairRedeemResp {
    pub token: String,           // full bearer "tnl_..."
    pub endpoint: String,        // echoed from daemon config
    pub name: String,            // for wizard display
}
```

**REST endpoints** (on tnld, served through Caddy/whatever reverse proxy):

| Method | Path                | Auth                  | Semantics                               |
|--------|---------------------|-----------------------|-----------------------------------------|
| POST   | `/pair`             | Bearer (admin scope)  | create pair-code for a new token        |
| POST   | `/pair/redeem`      | none (code is the credential) | exchange code for full bearer token |
| GET    | `/pair/list`        | Bearer (admin scope)  | list pending (unredeemed) codes         |

These are plain REST, not yamux. Called via `reqwest` from CLI before the
client has authenticated.

**Note on `invite_url`:** the `invite_url` field returned by `/pair` is a
**human-readable shareable string**, not an actual server route. It encodes
endpoint + code in URL form (`https://tnl-api.example.com/invite/AB-12-CD`)
so the user can copy a single string instead of two values. `tnl init` parses
this URL to extract the endpoint and the pair code, then POSTs the code to
`/pair/redeem` — the `/invite/...` path is never hit by the client. If a
browser follows the link, the server returns 404 — accepted trade-off for
beta; a friendly HTML landing page can come in rc.1 if user feedback
warrants it.

**Code format:**

Six characters from a 32-character unambiguous alphabet
(`ABCDEFGHJKMNPQRSTUVWXYZ23456789` — no 0/O, 1/I/l), formatted as `AB-12-CD`
for readability. Entropy: log2(32^6) ≈ 30 bits. Sufficient given the
5-minute TTL and rate limits below.

**Rate limiting** (server-side, in `PairRegistry`):

- 3 redeem attempts per IP per minute
- 100 redeem attempts globally per minute
- 5 wrong attempts against the same code → code invalidated (independent of
  TTL)
- Max 16 concurrently-live pending codes

Expected time to brute-force one code at the global rate of 100 req/min ≈
107 days. Within a 5-minute TTL this is unreachable.

**In-memory registry** (`crates/tnld/src/pair.rs`):

```rust
pub struct PairEntry {
    pub code:       String,
    pub name:       String,
    pub created_at: Instant,
    pub expires_at: Instant,
    pub attempts:   u8,
}

pub struct PairRegistry {
    inner:      DashMap<String, PairEntry>,
    by_ip:      governor::RateLimiter<IpAddr, ...>,
    global:     governor::RateLimiter<NotKeyed, ...>,
}
```

Persistence: none. Codes are lost on daemon restart — accepted trade-off.
Pairing is a setup-time action; rerunning `tnld token add laptop --pair` is
cheap.

**Failure codes** (returned by `/pair/redeem`):

| HTTP | Body                                        | Client message                                              |
|------|---------------------------------------------|-------------------------------------------------------------|
| 404  | `{"error": "pair_not_found"}`               | "invite expired or already used. Ask for a new one."        |
| 410  | `{"error": "pair_expired"}`                 | "invite expired. Ask for a new one."                        |
| 429  | `{"error": "pair_too_many_attempts"}`       | "this invite was invalidated. Ask for a new one."           |
| 429  | `{"error": "rate_limited", "retry_after": N}` | "too many attempts; wait Ns and retry."                   |
| 500  | `{"error": "internal"}`                     | "server failed to save token; please retry or contact admin." |

### Wizards (`tnl init`, `tnld init`)

Built on `dialoguer` (synchronous Rust prompt library). Not `ratatui`. Colors
via `nu-ansi-term`, gated by `NO_COLOR` / `--no-color` / `!stdout.is_terminal()`.

**`tnl init` scenarios:**

1. **By invite link** (most common):
   ```
   $ tnl init
   ? How would you like to authenticate?
     ▸ Use an invite link (recommended)
       Paste a token I already have
   ? Paste invite link: https://tnl-api.example.com/invite/AB-12-CD
     ✓ link parsed
   ? Redeem invite (one-time, expires in 4m 38s)? (Y/n) y
     ✓ paired as "laptop"
     ✓ endpoint reachable
   Saved configuration to ~/.config/tnl/config.toml (mode 0600).
   ? Install shell completion for zsh? (Y/n) y
     ✓ ~/.zfunc/_tnl  (add `fpath+=~/.zfunc` to .zshrc if not already)
   Done. Open a tunnel: tnl http 3000
   ```

2. **Manual paste** — fallback for when invite is impractical. Prompts for
   endpoint URL + hidden token input, validates with `/healthz` + `/whoami`.

3. **Existing config** — detects `~/.config/tnl/config.toml`, offers
   "test current connection" or "replace with new credentials".

4. **Non-interactive** — `--invite <url>`, `--endpoint + --token`, `--json`,
   `-y` flags; required for piped stdin. Fails fast with clear "missing
   required: ..." if not all values provided.

**`tnld init` scenarios:**

Runs inside the container (via `docker exec` or as part of
`tnld-install.sh`). Walks through `listen`, `public_url`, `hostname_root`,
`tokens_file` with sensible defaults. Optionally creates first admin token
+ invite. Prints a quick-start checklist showing what the user still needs
to do manually (reverse proxy, wildcard cert, wildcard A record) — this is
deliberate, the wizard does NOT touch Caddy/nginx/Traefik configs.

**What the wizards do NOT do:**

- Edit `/etc/caddy/...` or any reverse-proxy config — too many proxies,
  too many user setups
- Call DNS provider APIs (CF, Route53, etc.)
- Install Docker / open firewall ports
- Modify shell rc files — completion is printed with "add this line"
  guidance, never auto-appended

### Installers

**`scripts/install.sh`** (client, POSIX sh, ~300 LOC):

Usage:
```bash
curl -fsSL https://raw.githubusercontent.com/atassis/tnl/main/scripts/install.sh | sh
# overrides:
TNL_VERSION=v0.1.0-beta.1 sh ...
TNL_INSTALL_DIR=/usr/local/bin sh ...
```

Algorithm:
1. Detect OS + arch (`uname -s`, `uname -m`)
2. Resolve version (env or GitHub API `/releases/latest`)
3. Choose install dir (`TNL_INSTALL_DIR`, else `~/.local/bin`, else
   `/usr/local/bin` after sudo prompt)
4. Download tarball + `.sha256` from GitHub Releases, verify
5. Extract, `mv` binary, `chmod +x`
6. Post-install: print PATH guidance if dir not in PATH, suggest `tnl init`

Failure modes: missing `curl`/`tar`/`sha256sum` → exit with "please install:
..."; HTTP 404 on asset → list supported targets; checksum mismatch → exit;
install dir not writable + no sudo → exit with `TNL_INSTALL_DIR` guidance.

What it does NOT do: modify `.bashrc`/`.zshrc`, install shell completion
(deferred to `tnl init`), call `sudo` without explicit consent.

**`scripts/tnld-install.sh`** (server, Linux + Docker, ~500 LOC):

Usage:
```bash
curl -fsSL https://raw.githubusercontent.com/atassis/tnl/main/scripts/tnld-install.sh | sh
# non-interactive:
curl -fsSL ... | PUBLIC_URL=https://tnl-api.example.com HOSTNAME_ROOT=t.example.com NON_INTERACTIVE=1 sh
```

Algorithm:
1. Preflight: root or sudo, docker + compose plugin present, internet to
   ghcr.io
2. `docker pull ghcr.io/atassis/tnld:${TNL_VERSION:-latest}`
3. Create `/etc/tnld/` (root:tnld 0750)
4. Run `tnld init` wizard (or NON_INTERACTIVE env path)
5. Write `/opt/tnld/compose.yaml` with the canonical service definition
   (host network, read-only fs, rw mount for tokens, healthcheck)
6. `docker compose -f /opt/tnld/compose.yaml up -d`, wait for healthcheck
7. Create first admin token + invite (`docker exec tnld tnld token add admin --pair`)
8. Print invite link + quick-start checklist with manual items (reverse
   proxy, wildcard cert, DNS A record)

The container ships proxy snippet examples in
`/usr/share/doc/tnld/proxy-examples/` (caddy / nginx / traefik). The script
prints the path. Multi-DNS-provider docs live in `docs/dns-providers.md`.

Re-run safety: existing files prompt with default "keep" rather than
overwrite.

**`scripts/tnld-uninstall.sh`** removes the running compose service, the
image (optionally), and `/etc/tnld` + `/opt/tnld` (with confirmation).

### Single tnld binary, admin via subcommands

Rejected alternative: a separate `tnld-admin` binary. Reasons to keep single
binary: `tnld hash-password` is already a subcommand (pattern established),
all admin code needs the same `auth.rs` and `registry.rs`, one artifact
simplifies the Docker image.

Subcommands:
```
tnld serve --config /etc/tnld/config.toml      # daemon (existing)
tnld hash-password <plaintext>                  # existing
tnld init                                       # wizard
tnld healthcheck                                # for docker healthcheck:
tnld token add <name> [--pair]                  # create token (+ invite)
tnld token list                                 # show name, hash prefix, etc.
tnld token revoke <name>                        # remove + close sessions
```

`tokens.toml` is mounted **read-write** into the container so `tnld token
add` can write it; daemon picks up changes via `inotify` (already in spec).

Token names are unique. `tnld token add laptop` when a `laptop` entry
already exists → error ("token 'laptop' already exists; use --replace or
choose another name"). `tnld token revoke laptop` on a missing entry →
exit code 64, error "no such token: 'laptop'".

### Inspector (client-side)

Lives on the client because only the forwarder sees real bytes. The daemon
is a byte pump.

`crates/tnl/src/forwarder.rs` grows a `PeekingPump`:

```rust
pub struct PeekingPump {
    head_peek_req:  BytesMut,    // first 4 KB until \r\n\r\n
    head_peek_resp: BytesMut,    // first ~512 B
    bytes_in:       AtomicU64,
    bytes_out:      AtomicU64,
    started:        Instant,
    log_tx:         mpsc::Sender<LogLine>,
}
```

Peek terminates at first `\r\n\r\n` or 4 KB cap, then the pump switches to
pure byte-copy with no overhead.

`crates/tnl/src/inspector.rs` is the consumer task:

Default output (one line per request):
```
14:23:01.234  GET    /                       200    45ms     1.2.3.4     1.2KB
14:23:01.890  GET    /static/app.js          200    12ms     1.2.3.4    23.4KB
14:23:02.108  POST   /api/login              401     8ms     1.2.3.4     320B
```

Flags:
- `--quiet` — suppress all output except errors
- `-v` / `--verbose` — adds request and response headers
- `-vv` — adds first 4 KB of body as hex+ASCII
- `--json` — JSON lines for piping (`tnl http 3000 demo --json | jq`)
- `--log-file <path>` — also writes JSON lines to file
  (`$XDG_STATE_HOME/tnl/` default)

Status codes are colored on TTY (2xx green, 3xx blue, 4xx yellow, 5xx red).
Disabled by `NO_COLOR` / `--no-color`.

### `tnl status` / `tnl stop`

REST endpoints on tnld:

| Method | Path                       | Auth                | Semantics                          |
|--------|----------------------------|---------------------|------------------------------------|
| GET    | `/tunnels`                 | Bearer              | tunnels owned by this token        |
| GET    | `/tunnels?all=true`        | Bearer + admin      | all tunnels                        |
| DELETE | `/tunnels/{subdomain}`     | Bearer (owner or admin) | close tunnel + its session     |

`Tunnel` struct gains `owner_token_name: String` for filtering.

CLI:
```
$ tnl status
SUBDOMAIN    UPTIME    REQUESTS   BYTES IN   BYTES OUT   LOCAL
demo         12m 30s   42         15.2 KB    1.4 MB      127.0.0.1:9999

$ tnl stop demo
✓ tunnel "demo" closed (forwarded 1.4 MB over 42 requests in 12m 30s)
```

`--json` switches table to array-of-objects.

### Reconnect / reattach

**Daemon-side**:

```rust
pub enum SessionState {
    Active { handle: Arc<Mutex<dyn Session>> },
    Disconnected { since: Instant },
}

pub struct Tunnel {
    pub tunnel_id:   Ulid,
    pub subdomain:   String,
    pub hostname:    String,
    pub owner_token: String,
    pub state:       SessionState,
    pub created_at:  Instant,
    pub stats:       TunnelStats,
}
```

When a session dies (WS close, yamux EOF), daemon flips state to
`Disconnected { since: now }`. Background GC task (5s tick) removes tunnels
whose `since.elapsed() > session_grace_sec` (default 30s, configurable in
`[limits]`).

**Client-side** (`crates/tnl/src/reconnect.rs`):

Exponential backoff 1s → 2s → 4s → ... up to 30s, no max attempts. On each
new WS, client sends `ReattachReq { tunnel_id, subdomain }`. Daemon looks
up a matching `Disconnected` tunnel within grace, swaps session, replies
`ReattachResp { ok: true }`. If not found, replies
`Error { code: "tunnel_lost" }` and client falls back to
`CreateTunnelReq` with the same subdomain (which may now be claimed —
surface clearly).

UX:
```
$ tnl http 3000 demo
┌─ tnl ─────────────────────────────────────────
│ Tunnel:    https://demo.t.example.com
...

warning: connection lost (yamux: EOF); reconnecting in 1s
warning: connection lost (yamux: EOF); reconnecting in 2s
✓ reattached to existing tunnel "demo" (24s in grace window)
```

### Healthcheck subcommand

`tnld healthcheck` — connects to `cfg.listen + "/healthz"`, exit 0 on
2xx, exit 1 otherwise. Silent on stdout by default. Used by docker
compose `healthcheck:` so we don't depend on `curl` being in the alpine
runtime image.

### Random subdomain

`CreateTunnelReq.subdomain` becomes `Option<String>`. When `None`, server
generates an adjective-noun-N pattern (ngrok-style memorable):

```
happy-otter-12
swift-meadow-7
quiet-river-83
```

Wordlist (`crates/tnl-protocol/src/wordlists.rs`): 64 adjectives + 64
nouns, single-word lowercase, no profanity, numeric suffix 1..=99. Search
space ≈ 405k. On collision retry up to 5 times, then 503. Client CLI:

```
$ tnl http 3000               # → https://happy-otter-12.t.example.com
$ tnl http 3000 demo          # explicit; errors if taken
```

Closes a regression flagged during alpha shipping: the spec said
`[SUBDOMAIN]` (optional), the alpha implementation made it required.

## CLI hygiene pass (full)

### Output / display

- `--json` mode on every command with structured output; warnings/errors to
  stderr; pipe-friendly
- `--no-color` flag + `NO_COLOR` env (no-color.org spec)
- Auto-detect `!stdout.is_terminal()` → disable color, switch table → TSV
- `--verbose` = info, `-vv` = debug, `-vvv` = trace via `tracing-subscriber`
- `--quiet` suppresses everything except errors+warnings
- Box-drawing banner (`tnl http`) only in TTY; in pipe → plain text
- Emoji minimal (`✓ ✗ ▸`), ASCII fallback (`[ok] [err] [>]`) in non-UTF-8
  locales

### Exit codes (sysexits-style)

Stable; documented in `--help` footer and man pages. Considered part of
the CLI API for semver purposes.

```
0    success
1    generic failure
2    invalid usage (clap default)
64   config not found / not authenticated
65   token rejected
66   subdomain conflict / not available
67   local port unreachable
68   server unreachable (net/dns/tls)
70   server error (5xx)
75   pairing failure (expired / not found / rate limit)
130  Ctrl-C (SIGINT)
```

### Discoverability

- **Shell completions** via `clap_complete` for bash/zsh/fish/elvish/powershell.
  `tnl completion <shell>` writes to stdout; `tnl init` offers install.
- **Man pages** via `clap_mangen`, bundled in release tarballs under
  `man/man1/{tnl,tnld,tnld-token}.1`. Installer copies to
  `$XDG_DATA_HOME/man/man1/`.
- **`--help`** carries `# Examples` section in long_help.
- **`tnl doctor`** self-diagnostic: config exists? endpoint reachable? token
  valid? DNS resolves? cert valid? clock skew? Output:
  ```
  ✓ config at /home/user/.config/tnl/config.toml
  ✓ /healthz: 200 OK
  ✗ /whoami: 401 Unauthorized
       Token rejected. Run `tnl init` to re-authenticate.
  ⚠ clock skew 12s (server ahead)
  ```
- **`tnl version --json`** includes `commit`, `build_date`, `rust` via
  `built` crate.
- **Update notification** — daily background check against GitHub Releases
  API, cached in `$XDG_CACHE_HOME/tnl/update-check`. Opt-out via
  `--no-update-check` or `TNL_NO_UPDATE_CHECK=1`.

### Robustness

- `reqwest` calls — 10s connect timeout, 30s total
- Atomic config writes (tempfile + rename + chmod 0600)
- Graceful Ctrl-C via `tokio::signal::ctrl_c`, sends `Close` on control
  stream, exits 130
- Idempotent `tnl init`, `tnl auth login`, `tnld init`
- Argument validation up-front (subdomain regex, port range) before any
  network action
- Token never logged at info level; `tnl config show` masks
  (`tnl_abcd...••••`) unless `--show-secrets`

### Config paths (XDG-compliant)

|          | Client (tnl)                                              | Daemon (tnld)                                  |
|----------|-----------------------------------------------------------|------------------------------------------------|
| Config   | `$XDG_CONFIG_HOME/tnl/config.toml` → `~/.config/tnl/...`  | `/etc/tnld/config.toml` (prod); `$XDG_CONFIG_HOME/tnld/` (dev/macOS) |
| Cache    | `$XDG_CACHE_HOME/tnl/`                                    | n/a                                            |
| State    | `$XDG_STATE_HOME/tnl/`                                    | `/var/log/tnld/` or stdout                     |
| Override | `TNL_CONFIG=<path>`                                       | `--config <path>` or `TNLD_CONFIG`             |

### Stability commitments (in CONTRIBUTING.md)

- CLI: removing/renaming a flag or command → major bump
- `--json` schema: removing/renaming a field → major
- Exit codes: adding → minor; repurposing existing code → major
- Wire protocol: removing existing `ControlMsg` variant → major; adding
  variants → minor (forward-compat)

Pre-1.0 these are aspirational but we hold to them in beta to elicit
honest feedback.

### Packaging

- `cargo install tnl-cli` (the name `tnl` may already be on crates.io)
- `cargo install --git` works naturally
- `ghcr.io/atassis/tnld:<tag>` + `:latest` (non-prerelease) + `:edge` (main)
- Homebrew, apt, rpm — deferred to rc.1 / 1.0

|---------------|------------------------------------------------|-----------------------------------------------------|--------------------|
| Unit          | `crates/*/src/**/*.rs` under `#[cfg(test)]`    | parsers, validators, pair registry, decode_chunked, subdomain generator | PR + pre-commit |
| Integration   | `crates/<crate>/tests/*.rs`                    | in-process spawn-server, REST round-trips           | PR                 |

| PTY           | `crates/tnl/tests/wizard_*.rs` via `expectrl`  | interactive `tnl init`, `tnld init`                 | PR                 |

### New beta-specific tests

**Pairing** (`tnld/tests/pair_create_redeem.rs`):
- happy: create → redeem → 200 → second redeem → 404
- expiry: TTL 1s, sleep 2s, redeem → 410
- rate limit: 4× redeem wrong code from one IP → 429 on 4th
- 5× wrong → code invalidated
- admin scope: `/pair` without admin Bearer → 403

**Reconnect** (`tnl-e2e/tests/reconnect.rs`):
- forced WS close on server, client reattaches within grace window, same
  `tunnel_id`
- after grace expiry: fresh `CreateTunnel`, new `tunnel_id`
- blackhole server: client gives up after N attempts (configurable)

**Random subdomain** (`tnld/tests/random_subdomain.rs`):
- 100 calls → all match regex, no collisions in registry
- profanity check against 50-word forbidden list

**Inspector** (`tnl/tests/inspector_format.rs`):
- feed mock request bytes → method+path extracted
- mock response → status extracted
- `--json` schema validated via `jsonschema`

**Wizards** (`{tnl,tnld}/tests/wizard_*.rs` via `expectrl`):
- spawn binary, expect prompt, send input, assert side effects (config
  files, token files)

**Admin CLI** (`tnld/tests/admin_token.rs`):
- add → /whoami works
- list → JSON contains added name
- revoke → /whoami returns 401 within 200ms (inotify reload)

**Doctor** (`tnl/tests/doctor.rs`):
- all-green: mock tnld, every check ✓
- bad token: mock returns 401 on /whoami → "✗ /whoami: 401" with hint
- clock skew: feed `Date: <ts+600s>` → "⚠ clock skew 10m 0s"
- `--json` schema validated

**Healthcheck** (`tnld/tests/healthcheck_cmd.rs`):
- daemon up → exit 0
- daemon down → exit 1, stderr "connection refused"

### Shell tests (bats)

`scripts/tests/install_*.bats` — fresh install, reinstall idempotency,
custom dir, missing dependencies, unsupported OS.

`scripts/tests/tnld_install_*.bats` — docker-in-docker run on fresh
ubuntu; assert compose file written, tnld container healthy.

### Regression tests from §2 prod-deploy bugs

| Bug                                                                | Test                                                                                    |
|--------------------------------------------------------------------|-----------------------------------------------------------------------------------------|
| TLS feature off in client (alpha-deploy bug)                       | `tnl-e2e/tests/wss_dial.rs` — spawn rustls test proxy, tnl dials wss://, assert success |
| data_plane wrapped raw bytes as octet-stream (alpha-shipping bug)  | Already added in `crates/tnld/src/data_plane.rs` mod tests + e2e content-type asserts   |
| `/etc/tnld` 700 unreadable from container uid 100                  | `tnld/tests/file_perms.rs` — image runs as uid 100, reads 644-mode files                |
| Data-plane logs too sparse                                         | `tnld/tests/data_plane_logs.rs` — `RUST_LOG=tnld=debug`, assert host+method+status logged |
| `subdomain` required despite spec saying optional                  | Covered by random_subdomain test                                                        |

### Anti-flakiness rules

- No real internet calls in tests; mock everything (`axum` spawn, no
  `wiremock-rs`)
- Every `await` in integration tests wrapped in `tokio::time::timeout`
- No `sleep(100ms)` — use channel/event waits
- Single `tokio::test` runtime per test, no manual spawn
- Bind to `127.0.0.1:0` — kernel assigns port

### Coverage non-goals (documented in CONTRIBUTING.md)

- Real network conditions (loss, jitter) — manual only
- Cross-OS beyond Linux+macOS (Windows, *BSD) — best-effort, doesn't block
  CI
- Load testing (>100 concurrent tunnels) — separate bench suite in 1.0
- DNS provider API integration — docs only

### Pre-release smoke checklist

When tagging `v0.1.0-beta.X`:

1. Fresh VM (one Linux x86_64, one macOS aarch64): run `install.sh`
2. `tnl init`, pair with your-gateway daemon
3. `tnl http 3000` + browser test via real TLS / CDN path
4. `tnl doctor` → all green
5. `tnl status` shows the tunnel; `tnl stop` closes it
6. On server: `tnld token list` shows the newly-paired client
7. Record in `docs/release-checklist.md`, commit `verified v0.1.0-beta.X`

## Open questions / future work

These came up during brainstorming but are deferred:

- Should pair codes be persisted across daemon restart? Currently no.
  Reconsider if users hit "I had to restart docker and lost my codes" pain.
- Pairing endpoint should support QR codes for mobile-paired clients
  (v0.2 likely).
- `tnl share <file>` one-shot static file serve mode (would replace
  `python -m http.server` for many quick-share use cases).
- Admin scope semantics — currently binary (admin or not). Could become
  granular scopes (`tunnel:create`, `pair:create`, `tokens:write`) in 1.0.
- Self-update via `tnl self-update` subcommand (in addition to passive
  notification). Defer until installer feedback comes in.
