# tnl v0.1.0-alpha.3 — Error Attribution & Forwarder Robustness

**Status:** Design ratified 2026-05-27. Implementation deferred until v0.1.0-beta lands.
**Driver:** Field bug 2026-05-27 — Vite-style backend bound to `[::1]:5173` while
`tnl` forwarder connected only to `127.0.0.1`, producing an opaque
`502 bad backend response` from `tnld` instead of any usable diagnostic.
**Audience:** Same as alpha.1 — author + early friends. No public OSS expectations
yet; that begins at beta.

## 1. Problem

When an end user hits `https://<sub>.t.example.com/` and something fails between
Caddy and the developer's machine, the public response gives no clue about where
the failure occurred, and the developer's terminal gives no clue that the request
even arrived. Concretely, three classes of failure all collapse to the same
unattributed `502 bad backend response`:

1. **Local backend not reachable** — wrong stack (IPv6-only bind), wrong port,
   refused, timeout, hung process. End user sees 502. Developer sees nothing
   on the CLI. tnl is blamed for what is a local issue.
2. **`tnld` transport hiccup** — yamux session died, substream open failed,
   write/read errored. End user sees 502. Same wording as case 1.
3. **Client misbehavior** — client wrote garbage to the substream (alpha bug
   the user already hit). Same 502.

The forwarder's defect that triggered the field incident — connecting only to
the IPv4 loopback literal — is a single line, but the larger ergonomic problem
is that the system gives users zero feedback loop closure when their backend is
the cause. Fixing only the IPv4 literal would still leave every other local-
side failure (port hung, malformed response, no listener) producing the same
opaque 502.

## 2. Goals

1. End user receives a response that **clearly identifies where the failure
   occurred** and, when the failure is local-side, **what to fix and how**.
2. Developer's terminal shows **one comprehensible line per request** including
   resolved target, status, and (on failure) failure kind — proof the request
   completed the public-side path.
3. Forwarder no longer fails on IPv6-only backends (the original bug) and
   handles equivalent failure modes (connect timeout, EOF before response,
   malformed response, slow first byte) with attribution rather than silence.
4. `tnl http` accepts an optional explicit target (`<host>:<port>`,
   ngrok-style) for docker-bridge and LAN-host scenarios.
5. Daemon-side responses are semantically correct (404 for unregistered host,
   503 for temporarily-unavailable, 502 for transport-level bad gateway) with
   `X-Tnl-Component` headers identifying ownership of the failure.

## 3. Non-Goals

- **No reconnect / reattach window.** That is beta scope; if the session dies,
  alpha.3 still returns 503 immediately.
- **No `tnl doctor`, `tnl status`, `tnl stop`.** Beta.
- **No full inspector.** Alpha.2 ships a minimal `mpsc::Sender<LogLine>`-based
  stdout logger; beta replaces the consumer with a colored, peek-aware TUI.
- **No metrics / Prometheus endpoint.** Beta or later.
- **No `--allow` / `--auth` / rate-limit features.** Beta (deferred) or 1.0.
- **No URL-form targets (`https://...`, `file://`).** Beta+; TLS upstream and
  static file serving are separate concerns.
- **No `--host` flag.** Custom target uses positional `<host>:<port>` syntax,
  matching ngrok.

## 4. Design

### 4.1 Architecture overview

Two-side cooperation: the forwarder on the CLI synthesizes the response when
it knows the failure is local-side, and the daemon's `data_plane.rs` is
tightened to emit attribution headers for the failure modes it owns.

```
            ┌─────────────────────────────────── tnld (data_plane.rs) ────────────────┐
end-user ──►│  Host lookup ──┬─► 404 + X-Tnl-Component: registry (no tunnel)         │
            │                ├─► 503 + X-Tnl-Component: daemon  (session not ready)  │
            │                └─► open_stream + write head ──┐                        │
            │                                               │                        │
            │                              ┌────────────────┴───────────────┐        │
            │                              │ substream → ── proxy bytes ─── │ ◄──┐   │
            │                              │ peek response first line ─────►│    │   │
            │                              │ 502 + X-Tnl-Component: client  │    │   │
            │                              │   (malformed response)         │    │   │
            │                              │ 502 + X-Tnl-Component: trans   │    │   │
            │                              │   (substream errored mid)      │    │   │
            │                              └────────────────────────────────┘    │   │
            └─────────────────────────────────────────────────────────────────┼──┘
                                                                              │
                                                                              ▼
            ┌────────────────────────────────── tnl forwarder ────────────────────┐
            │ peek request head ─► parse Accept                                   │
            │                                                                     │
            │ connect_local(target) ─┬─► ok ──► copy_bidirectional                │
            │                        │                                            │
            │                        └─► err ─► synth_response(kind, accept) ──── │ → substream
            │                                                                     │
            │ probe response first line ──┬─► looks like HTTP ─► copy through    │
            │                             │                                       │
            │                             └─► EOF/garbage/timeout ─► synth + close│ → substream
            │                                                                     │
            │ on every request: send LogLine via mpsc::Sender                     │
            └─────────────────────────────────────────────────────────────────────┘
```

### 4.2 Wire-level signaling

The CLI **synthesizes a complete HTTP/1.1 response** and writes it to the yamux
substream whenever it detects local-side failure before any backend response
bytes have been forwarded. No protocol change. `tnld` parses this response
exactly as it would parse one from a real backend.

The window during which synthesis is possible:

- Phase 1 (pre-connect): CLI has the request head buffered, has not opened
  TCP yet. Full synthesis available.
- Phase 2 (post-connect, pre-first-byte): CLI has connected, written the head
  and any buffered body, but received zero bytes back from the backend.
  Synthesis still available; backend connection closed without sending to it.
- Phase 3 (mid-response): CLI has already forwarded some bytes from the
  backend into the substream. Synthesis no longer possible — the upstream
  HTTP framing is in progress. This case is acceptable as a corner; beta
  inspector can surface it through logs.

### 4.3 Failure kind catalog

| Kind                  | Phase | Cause                                                         |
|-----------------------|-------|---------------------------------------------------------------|
| `connect-refused`     | 1     | All resolved addresses returned `ECONNREFUSED`                |
| `connect-timeout`     | 1     | No address completed connect within `LOCAL_CONNECT_TIMEOUT`   |
| `connect-unreachable` | 1     | `EHOSTUNREACH`/`ENETUNREACH`/`EAFNOSUPPORT` on all addresses  |
| `dns-failed`          | 1     | `lookup_host` returned an error or zero addresses             |
| `local-no-response`   | 2     | TCP connected, request written, no response byte within timeout |
| `local-eof`           | 2     | TCP connected, request written, EOF before first response byte  |
| `local-malformed`     | 2     | Backend sent bytes, but they don't begin with `HTTP/1.` or `HTTP/2` |

Constants (hardcoded in alpha.3; made configurable in beta `[limits]`):

```rust
const LOCAL_CONNECT_TIMEOUT: Duration   = Duration::from_secs(3);
const LOCAL_FIRST_BYTE_TIMEOUT: Duration = Duration::from_secs(15);
const RESPONSE_PROBE_BYTES: usize = 9; // enough for "HTTP/1.0\0" or "HTTP/2\0"
const REQUEST_HEAD_PROBE_MAX: usize = 8 * 1024;
```

### 4.4 Synthesized response format

For any kind in §4.3, the CLI writes the following HTTP/1.1 message into the
substream:

```
HTTP/1.1 502 Bad Gateway\r\n
Content-Type: <negotiated>\r\n
Cache-Control: no-store\r\n
Connection: close\r\n
X-Tnl-Component: client\r\n
X-Tnl-Origin-Failure: <kind>\r\n
X-Tnl-Origin-Target: <resolved-host>:<port>\r\n
X-Tnl-Tunnel: <subdomain>\r\n
X-Tnl-Request-Id: <ulid>\r\n
Content-Length: <n>\r\n
\r\n
<body>
```

**Content negotiation** by `Accept` (parsed from the request head before
connect):

| Accept matches                          | Content-Type                  | Body |
|-----------------------------------------|-------------------------------|------|
| `text/html`, `*/*` from browsers (UA-sniff `Mozilla`) | `text/html; charset=utf-8` | HTML page (§4.5) |
| `application/json`, `application/*json` | `application/json`            | JSON (§4.5) |
| anything else, missing Accept, `*/*` not from browser | `text/plain; charset=utf-8` | One-line text (§4.5) |

### 4.5 Response body templates

All three body formats include: subdomain, resolved target, failure kind,
human-readable reason, and request ID.

**HTML** (one-segment design target ≤ ~1.4 KB; soft guideline, not enforced):

```html
<!doctype html><html lang="en"><head><meta charset="utf-8">
<title>tnl: local backend unreachable</title>
<style>body{font:14px/1.4 system-ui,sans-serif;max-width:36em;margin:3em auto;padding:0 1em;color:#222}
code{background:#f4f4f4;padding:.1em .3em;border-radius:.2em}
.fix{border-left:3px solid #c33;padding:.5em 1em;margin:1em 0;background:#fff5f5}
.foot{color:#888;font-size:.85em;margin-top:2em}</style></head><body>
<h1>Local backend not reachable</h1>
<p>Tunnel <code>{{tunnel}}</code> reached the developer's machine, but the
local server at <code>{{target}}</code> did not respond.</p>
<p><b>Reason:</b> {{reason}} <code>({{kind}})</code></p>
<div class="fix">{{hint}}</div>
<p class="foot">tnl v{{version}} · request <code>{{req_id}}</code></p>
</body></html>
```

**Plain text:**

```
tnl: local backend at {{target}} did not respond ({{kind}}: {{reason}}).
Tunnel: {{tunnel}}
Hint:   {{hint_oneline}}
Request: {{req_id}}
```

**JSON:**

```json
{
  "error": "local_backend_failure",
  "kind": "connect-refused",
  "target": "127.0.0.1:5173",
  "tunnel": "demo",
  "reason": "Connection refused on all resolved addresses",
  "hint": "Is your dev server running and bound to the right interface?",
  "request_id": "01J..."
}
```

**Hints by kind** (kept short — one line each):

| Kind                  | Hint                                                                 |
|-----------------------|----------------------------------------------------------------------|
| `connect-refused`     | Is your dev server running? For Vite/uvicorn try `--host 127.0.0.1`. |
| `connect-timeout`     | Server might be firewalled or unresponsive on `<target>`.            |
| `connect-unreachable` | Network stack unreachable; check `<target>` is correct.              |
| `dns-failed`          | DNS resolution failed; if target is custom, verify hostname.         |
| `local-no-response`   | Backend accepted the connection but did not send a response.         |
| `local-eof`           | Backend closed the connection without sending a response.            |
| `local-malformed`     | Backend sent data that is not a valid HTTP response.                 |

### 4.6 Size budget

Design target: full response (status line + headers + body) ≤ **1400 B** so it
fits in a single TCP segment on the end-user link after TLS+HTTP/2 framing.

This is a **soft guideline**, not enforced. Headers alone use ~350 B, leaving
~900 B for body. The three body formats above all fit. A regression sanity test
asserts each formatted response is ≤ 4 KB — catches obvious bloat (CSS
explosion, multi-paragraph copy) without blocking legitimate fixes that nudge
slightly over 1400 B.

### 4.7 Forwarder changes (`crates/tnl/src/forwarder.rs`)

Two-phase forward, replacing the current single-line `copy_bidirectional`:

```rust
pub struct ForwardCtx {
    pub tunnel: String,
    pub req_id: Ulid,
    pub log_tx: mpsc::Sender<LogLine>,
}

pub async fn forward(
    substream: Pin<Box<dyn Stream>>,
    target: Target,
    ctx: ForwardCtx,
) -> Result<()>;
```

Where `Target` is:

```rust
pub enum Target {
    LocalhostPort(u16),         // tnl http 5173 demo
    Explicit(SocketAddr),       // tnl http 127.0.0.1:5173 demo
}
```

Resolution rules:

- `LocalhostPort(port)` → `lookup_host(("localhost", port)).await` →
  `Vec<SocketAddr>` containing both `127.0.0.1:port` and `[::1]:port` from
  `/etc/hosts`. Iterate in returned order, first successful connect wins.
- `Explicit(addr)` → single-address vec, no DNS, no fallback.

`connect_local` is exported because tests exercise it directly:

```rust
pub(crate) async fn connect_local(target: &Target)
    -> Result<TcpStream, ConnectError>;

pub enum ConnectError {
    Refused,
    Timeout,
    Unreachable,
    DnsFailed(io::Error),
    DnsEmpty,
}
impl ConnectError {
    pub fn to_kind(&self) -> Kind { ... }
}
```

Phase 2 helpers:

```rust
async fn peek_request_head(substream: &mut Stream) -> Result<Bytes>;
async fn peek_response_first_line(tcp: &mut TcpStream, max: usize, timeout: Duration)
    -> Result<ResponseProbe>;

pub enum ResponseProbe {
    LooksHttp(Bytes),  // prefix to forward through unchanged
    Eof,
    NotHttp(Bytes),    // prefix for hex preview in synth message
    Timeout,
}
```

Naming convention: `PeekingPump` is reserved for the beta inspector's full
peek+pump primitive. Alpha.2 uses smaller `peek_request_head` /
`peek_response_first_line` helpers that the beta `PeekingPump` will subsume
without API thrash on consumers.

### 4.8 CLI positional arg parsing (`tnl http <target> <subdomain>`)

`<target>` accepts:

| Input            | Parsed as                             |
|------------------|---------------------------------------|
| `5173`           | `Target::LocalhostPort(5173)`         |
| `127.0.0.1:5173` | `Target::Explicit(127.0.0.1:5173)`    |
| `[::1]:5173`     | `Target::Explicit([::1]:5173)`        |
| `192.168.1.50:8080` | `Target::Explicit(192.168.1.50:8080)` |
| `foo:bar` (invalid) | Parse error with hint                |

Parsing via a clap `value_parser` that first tries `u16`, then
`SocketAddr::from_str`. Hostname forms (`backend.local:8080`) deferred to beta
— alpha.3 stays IP-only for explicit targets, since hostnames raise resolution-
order and trust questions worth deciding deliberately.

Help text update:

```
USAGE: tnl http <TARGET> <SUBDOMAIN>

ARGS:
  <TARGET>     Port (forwards to localhost) or host:port
                 Examples: 3000, 127.0.0.1:3000, 192.168.1.50:80, [::1]:8080
  <SUBDOMAIN>  Subdomain to expose under t.example.com
```

### 4.9 Daemon attribution (`crates/tnld/src/data_plane.rs`)

Status code matrix:

| Situation                                  | Status | `X-Tnl-Component` | Notes                              |
|--------------------------------------------|--------|-------------------|------------------------------------|
| Host not in registry                       | 404    | `registry`        | "no tunnel registered for <host>"  |
| Tunnel exists, no live session handle      | 503    | `daemon`          | `Retry-After: 1`                   |
| `open_stream` failed (yamux dead)          | 502    | `transport`       | Logged as daemon-side 5xx          |
| Write head/body to substream failed        | 502    | `transport`       | Logged as daemon-side 5xx          |
| Substream EOF before valid HTTP response   | 502    | `transport`       | Should be rare after alpha.3       |
| Backend response malformed (parse failed)  | 502    | `client`          | After alpha.3 ships, only seen from old CLI versions talking to new daemon; logged loudly as client misbehavior |
| Body read from end-user failed             | (terminate, no response) | n/a | End-user already disconnected      |

All daemon responses include:

- `X-Tnl-Component: <component>`
- `Cache-Control: no-store`
- `Connection: close`
- `Content-Type` per Accept negotiation (HTML / JSON / plain text)

Body rendering shared with CLI via a new module:

```rust
// crates/tnl-protocol/src/error_page.rs
pub enum Component { Registry, Daemon, Transport, Client }

pub struct ErrorContext<'a> {
    pub component: Component,
    pub kind:      Option<&'a str>,
    pub tunnel:    Option<&'a str>,
    pub target:    Option<&'a str>,
    pub reason:    &'a str,
    pub hint:      &'a str,
    pub req_id:    Ulid,
    pub version:   &'static str,
}

pub fn render(ctx: &ErrorContext<'_>, accept: Accept) -> Vec<u8>;
```

Both CLI and daemon call `error_page::render` with appropriate context.

### 4.10 Daemon-side 5xx as operational signal

Every daemon-emitted 5xx with `X-Tnl-Component ∈ {daemon, transport}` is logged
via `tracing::warn!`:

```rust
warn!(
    target: "tnl::data_plane::server_failure",
    component = ?ctx.component,
    method    = %method,
    path      = %path,
    tunnel    = %tunnel_name,
    req_id    = %req_id,
    "served {status} from daemon",
);
```

Grep-friendly target lets `journalctl -u tnld -g 'server_failure'` find them.
Counter / metrics endpoint deferred to beta.

### 4.11 CLI stdout logger (`crates/tnl/src/progress.rs`, new)

Channel-based, designed so beta inspector replaces only the consumer task.

```rust
// in tnl-protocol::messages (shared so beta can extend without forking)
//
// Types: Ulid = ulid::Ulid, Bytes = bytes::Bytes, SystemTime/Duration from std.
pub struct LogLine {
    pub req_id:        Ulid,
    pub ts:            SystemTime,
    pub method:        Bytes,
    pub path:          Bytes,
    pub display_target: String,           // "localhost:5173" or "127.0.0.1:5173" as typed
    pub resolved_addr: Option<SocketAddr>,// None until connect attempted; Some(last_tried) after
    pub status:        Option<u16>,
    pub bytes_out:     Option<u64>,
    pub duration:      Option<Duration>,
    pub failure_kind:  Option<String>,    // Kind serialized as &str (see §4.3)
    // beta-only fields, default None in alpha.3:
    pub peek_req_head:  Option<Bytes>,
    pub peek_resp_head: Option<Bytes>,
}
```

Producer: `ForwardCtx::log_tx`, which is a
`tokio::sync::mpsc::Sender<LogLine>` with a buffer of 256 (bounded so that a
stuck consumer back-pressures the forwarder rather than memory-balloons).
The forwarder emits two events per request: one when the request head is
parsed (`status: None`), one on completion (`status: Some(...)`).

If the channel is full when emitting, the forwarder drops the log event
silently and increments a `tracing::warn!` counter — the per-request log is
diagnostic, not load-bearing, and must never block byte forwarding.

Consumer in alpha.3 (`progress.rs`):

```rust
pub fn spawn_console_logger(rx: Receiver<LogLine>, quiet: bool) -> JoinHandle<()>;
```

A single task drains the channel and prints one line per *completed* request
(skips the start event):

```
14:23:01.234  GET    /                          → [::1]:5173       200   45ms     1.2 KB
14:23:01.890  POST   /api/login                 → 127.0.0.1:5173   502    3ms     connect-refused
14:23:02.108  GET    /static/very-long-path...  → [::1]:5173       200   12ms    23.4 KB
```

Column rules:

- Time `HH:MM:SS.mmm`
- Method, right-padded to 6 chars
- Path, truncated to 26 chars with `…`
- `→ <target>` resolved address, padded to 15 chars
- Status code (or `---` if no response was synthesized)
- Duration in ms or s
- Either `<bytes formatted>` (1.2 KB, 23.4 MB, ...) or the failure kind

No color. Beta inspector adds color, `--verbose` peeking, `--json`,
`--log-file`. `--quiet` suppresses per-request lines but keeps the start
banner and fatal errors.

## 5. Tests

Listed by file. Each `tests/` file is a `#[tokio::test]`-based integration
test using small mock backends.

### `crates/tnl/tests/`

- `forwarder_dual_stack_ipv4.rs` — IPv4-only `TcpListener::bind("127.0.0.1:0")`,
  forward target is `Target::LocalhostPort(port)`; assert request roundtrips
  and 200 returned.
- `forwarder_dual_stack_ipv6.rs` — IPv6-only `TcpListener::bind("[::1]:0")`,
  same target. The regression test for the field bug.
- `forwarder_explicit_ipv4.rs` — backend on `127.0.0.1`, target
  `Target::Explicit("127.0.0.1:NNNN".parse())`.
- `forwarder_explicit_ipv6.rs` — backend on `[::1]`, target
  `Target::Explicit("[::1]:NNNN".parse())`.
- `forwarder_connect_refused.rs` — pick a port with nothing on it; expect
  synth response with `X-Tnl-Origin-Failure: connect-refused`,
  `X-Tnl-Component: client`, status 502.
- `forwarder_connect_timeout.rs` — `iptables -j DROP` is not portable; use a
  black-hole IP (`192.0.2.0/24` is documentation range) with a short timeout
  override. Expect `connect-timeout` kind.
- `forwarder_local_eof.rs` — listener accepts then immediately closes; expect
  `local-eof` kind.
- `forwarder_local_malformed.rs` — listener accepts and writes
  `"hello not http garbage\r\n\r\n"`; expect `local-malformed` kind.
- `forwarder_local_no_response.rs` — listener accepts and sleeps past the
  first-byte timeout (override timeout in test); expect `local-no-response`.
- `forwarder_target_parsing.rs` — `Target::from_str` cases: `5173`,
  `127.0.0.1:5173`, `[::1]:5173`, `1.2.3.4:80`, plus negative
  cases.
- `progress_format.rs` — render a few `LogLine`s and assert column shape.

### `crates/tnld/tests/`

- `data_plane_503_no_session.rs` — register tunnel but no session handle;

## 6. Execution timing

**Implementation is deferred until v0.1.0-beta is merged.**

Rationale: the beta worktree is actively rewriting `forwarder.rs` (PeekingPump),
`data_plane.rs` (structured logging, task 5), and adding `LogLine` /
`--quiet` / inspector consumer. Executing alpha.3 in parallel produces a
guaranteed multi-file conflict on the most fragile crate in the project.

Upon beta merge:

1. Rebase this spec and its plan against current `main`.
2. Audit which alpha.3 features beta absorbed verbatim (likely:
   `PeekingPump` ≈ `peek_request_head`+`peek_response_first_line`, the
   `LogLine` struct, `--quiet`).
3. Execute remaining tasks — primarily synth-response rendering, daemon
   status code matrix, `error_page` module, `<target>` parsing, dual-stack
   resolution, and the test matrix.
4. Tag `v0.1.0-alpha.3` and ship.

If, by the time beta is merged, the spirit of this spec has been absorbed
into beta (the field bug fixed, attribution headers present, dual-stack
working), drop alpha.3 entirely and roll its remaining items into the v0.1.0
release prep.

## 7. Open questions

- **Hostname targets (`backend.local:8080`)** — alpha.3 stays IP-only for
  explicit targets. Whether to add DNS-resolvable hostnames for explicit
  targets is a beta decision (involves cache TTL, trust boundaries, and
  resolution-order semantics).
- **Per-tunnel timeout overrides** — `LOCAL_CONNECT_TIMEOUT` and
  `LOCAL_FIRST_BYTE_TIMEOUT` are hardcoded in alpha.3. Beta `[limits]`
  config should expose them.
- **Color in CLI logger** — alpha.3 ships colorless. If `tnl-protocol`-level
  changes from beta land before alpha.3 execution starts, this may become
  trivial to add early.
- **Mid-response failure** — Phase 3 in §4.2 is acknowledged as unhandled.
  Beta inspector will log it; alpha.3 simply lets the partial response reach
  the end user, which is the existing behavior.
